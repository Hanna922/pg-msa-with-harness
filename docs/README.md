# Payment Gateway MSA

## 프로젝트 소개

이 프로젝트는 카드 결제 승인 처리의 전체 흐름을 MSA(Microservice Architecture)로 구현한 결제 게이트웨이 시스템입니다. 고객이 카드로 결제를 시도하면 가맹점에서 PG사로, PG사에서 카드사로, 카드사에서 은행으로 승인 요청이 전달되고, 그 응답이 역순으로 돌아오는 과정을 7개의 독립 서비스로 분리하여 구현했습니다.

프로젝트의 핵심 목표는 단순 프록시 수준의 PG 서비스를 실무 수준의 결제 게이트웨이로 단계적으로 고도화하는 것입니다. 거래 이력을 기록하고, 중복 승인을 방지하며, 외부 카드사 통신 장애에 대응하고, 카드 브랜드에 따라 적절한 카드사로 라우팅하는 기능을 6단계에 걸쳐 설계하고 구현했습니다.

## 왜 이 프로젝트를 만들었는가

실제 결제 시스템은 단순히 API를 호출하고 응답을 전달하는 것이 아닙니다. 결제는 실패할 수 있고, 같은 요청이 두 번 올 수 있으며, 카드사 서버가 응답하지 않을 수도 있습니다. 이런 상황에서 돈이 이중으로 빠져나가거나, 거래 기록이 유실되거나, 하나의 카드사 장애가 전체 시스템을 멈추게 하는 일이 없어야 합니다.

이 프로젝트는 그런 문제들을 하나씩 해결해 나가는 과정 자체를 목적으로 합니다. 처음에는 요청을 받아서 그대로 전달하기만 하는 서비스에서 출발하여, 실무에서 PG사가 반드시 갖추어야 할 기능들을 단계적으로 추가했습니다. 각 단계는 이전 단계의 한계를 인식하고, 그것을 해결하기 위한 구조적 결정을 내린 결과물입니다.

## 시스템 구조

전체 시스템은 다음과 같은 흐름으로 결제 요청을 처리합니다. 외부 클라이언트의 결제 요청은 API Gateway를 거쳐 가맹점 서비스에 도착하고, 가맹점 서비스는 이를 PG 서비스에 전달합니다. PG 서비스는 카드 번호를 분석하여 적절한 카드사에 승인 요청을 보내고, 카드사는 카드 유효성을 검증한 뒤 체크카드인 경우 은행에 잔액 조회와 출금을 요청합니다.

이 중에서 PG 서비스만이 고도화 대상입니다. 가맹점, 카드사, 은행 서비스는 실제 외부 기관의 역할을 시뮬레이션하는 테스트 스텁이며, PG 서비스가 이들과 어떻게 안정적으로 통신하고 거래를 관리하는지를 검증하기 위해 존재합니다.

## 고도화 과정

### 거래 원장과 멱등성

PG가 가장 먼저 갖추어야 할 것은 거래를 기억하는 능력입니다. 초기 상태에서 PG 서비스는 데이터베이스 없이 요청을 받아 카드사에 전달하고 응답을 돌려주기만 했습니다. 이 구조에서는 어떤 거래가 언제 처리되었는지 알 수 없고, 같은 결제 요청이 두 번 들어오면 두 번 모두 카드사에 전달되어 이중 승인이 발생할 수 있습니다.

이를 해결하기 위해 거래 원장을 도입했습니다. 모든 승인 요청은 외부 카드사를 호출하기 전에 먼저 원장에 PENDING 상태로 기록됩니다. 카드사 응답이 돌아오면 APPROVED, FAILED, TIMEOUT 중 하나로 상태가 갱신됩니다. 가맹점이 동일한 거래 ID로 재요청을 보내면, 원장에서 기존 거래를 찾아 그 결과를 그대로 반환합니다. 카드사를 다시 호출하지 않으므로 이중 승인이 원천적으로 차단됩니다.

카드 번호는 원장에 그대로 저장하지 않습니다. 앞 6자리(발급 은행 식별 번호)와 뒤 4자리만 남기고 나머지를 마스킹하여 저장합니다. 이는 PCI DSS 관점에서 민감 정보 노출을 최소화하기 위한 조치입니다.

### 승인 흐름의 책임 분리

거래 원장이 도입되면서 PG 서비스의 승인 처리 흐름이 복잡해졌습니다. 입력 검증, 멱등성 확인, 원장 기록, 카드사 요청 조립, 외부 호출, 응답 매핑까지 모든 것이 하나의 서비스 클래스에 모여 있으면 유지보수가 어렵고 테스트도 힘듭니다.

이 문제를 해결하기 위해 승인 흐름을 여러 전담 컴포넌트로 분리했습니다. 입력 검증은 검증 전담 서비스가, 거래 원장 관리는 원장 서비스가, DTO 변환은 매퍼와 팩토리가, 이 모든 것의 조합은 Facade가 담당합니다. 컨트롤러는 HTTP 요청을 받아 Facade에 위임하고 결과를 반환하는 얇은 어댑터 역할만 합니다.

이 분리의 핵심은 각 컴포넌트가 자신의 책임만 알면 된다는 것입니다. 검증 서비스는 원장이 어떻게 동작하는지 모르고, 원장 서비스는 카드사를 어떻게 호출하는지 모릅니다. 덕분에 각 컴포넌트를 독립적으로 테스트할 수 있고, 한 부분의 변경이 다른 부분에 영향을 주지 않습니다.

### 외부 카드사 연동의 안정화

PG 서비스에서 가장 불안정한 구간은 외부 카드사와의 통신입니다. 네트워크 지연, 카드사 서버 장애, 일시적 오류 등 다양한 실패 상황이 발생할 수 있습니다. 이런 상황에 대한 대비 없이는 카드사 하나가 느려지면 PG 서비스 전체가 함께 느려지는 문제가 생깁니다.

이를 체계적으로 해결하기 위해 세 가지 방어 메커니즘을 순서대로 도입했습니다.

먼저 외부 호출에 타임아웃을 설정했습니다. 카드사가 응답하지 않을 때 무한정 기다리는 것을 방지하기 위해, 연결 타임아웃과 읽기 타임아웃을 설정 파일로 외부화하여 관리합니다.

다음으로, 일시적 통신 장애에 대한 재시도 정책을 추가했습니다. 다만 모든 실패를 재시도하지는 않습니다. 카드사가 의도적으로 거절한 경우(잔액 부족, 카드 정지 등)를 재시도하는 것은 의미가 없으므로, 순수한 통신 장애인 경우에만 제한된 횟수만큼 재시도합니다. 이를 위해 외부에서 발생하는 다양한 예외를 통신 장애, 카드사 측 실패, 빈 응답, 서킷 오픈의 네 가지 유형으로 분류하는 오류 번역 계층을 도입했습니다.

마지막으로 Circuit Breaker를 도입했습니다. 카드사가 지속적으로 실패하면 더 이상 요청을 보내지 않고 즉시 실패를 반환하여, 장애가 PG 서비스 전체로 전파되는 것을 차단합니다. 일정 시간이 지나면 자동으로 소수의 요청을 다시 보내보고, 성공하면 정상 상태로 복귀합니다.

이 과정에서 거래 원장의 상태 분류도 정교해졌습니다. 통신 타임아웃으로 카드사 응답을 받지 못한 거래는 TIMEOUT으로, 카드사가 응답은 했지만 시스템 오류를 반환한 거래는 FAILED로 구분하여 기록합니다. 이 구분이 있어야 이후 장애 분석 시 "카드사가 응답을 안 한 것"과 "카드사가 오류를 응답한 것"을 분리하여 파악할 수 있습니다.

### 예외 처리의 표준화

여러 계층에서 다양한 예외가 발생할 수 있는데, 이를 일관된 규칙 없이 각 컨트롤러에서 개별적으로 처리하면 응답 형식이 제각각이 되고, 새로운 예외 유형이 추가될 때마다 모든 컨트롤러를 수정해야 합니다.

이를 해결하기 위해 PG 서비스 내부의 모든 예외를 표준화했습니다. 먼저 오류 분류표를 정의하여 요청 검증 실패, 카드사 통신 장애, 카드사 측 실패, Circuit Breaker 오픈, 내부 오류의 다섯 가지로 분류하고, 각 분류에 HTTP 상태 코드를 매핑했습니다. 입력이 잘못되면 400, 카드사와 통신이 안 되면 503, 카드사가 오류를 반환하면 502, Circuit Breaker가 열려 있으면 429, 예상치 못한 오류는 500을 반환합니다.

이 분류에 따라 모든 예외를 하나의 표준 예외 타입으로 감싸고, 전역 예외 처리기가 이를 가로채어 일관된 응답을 생성합니다. 컨트롤러에서는 try-catch를 완전히 제거하고 DTO만 반환하는 순수한 HTTP 어댑터 역할만 남겼습니다.

### 라우팅 추상화

앞선 단계들에서 PG 서비스는 하나의 카드사만 연동하고 있었습니다. 그러나 실제 PG사는 여러 카드사와 연동하며, 카드 종류에 따라 적절한 카드사로 거래를 라우팅해야 합니다. 이를 위해 승인 흐름에서 "어느 카드사에 보낼 것인가"를 결정하는 라우팅 책임을 분리했습니다.

이 단계에서는 실제 다중 카드사 연동을 구현하지 않고, 구조적 기반만 마련했습니다. 라우팅 정책을 인터페이스로 추상화하고, 현재는 항상 동일한 카드사를 반환하는 기본 구현만 두었습니다. 승인 Facade가 카드사 클라이언트를 직접 호출하는 대신 라우팅 서비스를 통해 호출하도록 의존 방향을 변경한 것이 핵심입니다. 이렇게 하면 이후 라우팅 정책 구현체만 교체하여 다중 카드사 라우팅을 도입할 수 있습니다.

### BIN 기반 다중 카드사 라우팅

라우팅 추상화를 실제로 활용하는 단계입니다. 카드 번호의 앞자리(BIN, Bank Identification Number)를 분석하여 카드 브랜드를 식별하고, 브랜드에 따라 적절한 카드사로 승인 요청을 라우팅합니다. 이 프로젝트에서는 Visa 카드는 Acquirer A로, Mastercard, Amex, JCB, Discover 카드는 Acquirer B로 라우팅하는 규칙을 적용했습니다.

실무에서 BIN 기반 라우팅은 PG사의 핵심 기능 중 하나입니다. 같은 가맹점에서 발생한 결제라도 Visa 카드와 Mastercard 카드는 서로 다른 카드사를 통해 처리될 수 있으며, 어떤 카드사를 선택하느냐에 따라 수수료와 승인율이 달라집니다. 이 프로젝트에서는 BIN의 첫 번째 자리(또는 두 자리)만으로 브랜드를 식별하는 간소화된 규칙을 사용하지만, 실무에서는 BIN 6~8자리를 활용한 더 정밀한 라우팅 테이블을 운영합니다.

이 기능의 구현에서 중요한 설계 결정은 세 가지입니다. 첫째, BIN 식별과 라우팅 정책을 분리했습니다. 카드 번호에서 브랜드를 식별하는 것과, 그 브랜드를 어느 카드사로 보낼지 결정하는 것은 다른 책임입니다. BIN 식별기는 라우팅뿐 아니라 사기 탐지나 통계 등 다른 맥락에서도 재사용할 수 있습니다. 둘째, 브랜드-카드사 매핑을 설정 파일로 외부화했습니다. 코드를 변경하지 않고도 특정 브랜드의 라우팅 대상을 바꿀 수 있습니다. 셋째, 각 카드사 클라이언트가 독립된 Circuit Breaker 인스턴스를 갖도록 했습니다. Acquirer A에 장애가 발생해도 Acquirer B의 처리에는 영향이 없습니다.

두 번째 카드사를 추가하면서 공통 클라이언트 인터페이스를 도입했고, 라우팅 서비스가 이 인터페이스 목록을 자동으로 수집하여 카드사 유형별로 적절한 클라이언트를 선택합니다. 이 구조 덕분에 세 번째 카드사를 추가할 때 라우팅 서비스의 코드를 수정할 필요 없이, 새 클라이언트 구현과 설정 추가만으로 확장이 가능합니다.

거래 원장에도 어느 카드사를 경유했는지를 기록합니다. 이 정보는 향후 카드사별 정산, 승인율 분석, 장애 영향도 파악의 기초 데이터가 됩니다.

## 기술 스택

이 프로젝트는 Spring Boot와 Spring Cloud를 기반으로 합니다. 서비스 간 통신은 OpenFeign 클라이언트와 Eureka 서비스 디스커버리를 사용하며, 외부 연동의 안정성을 위해 Resilience4j Circuit Breaker를 적용했습니다. PG 서비스의 거래 원장은 JPA로 관리하며, 로컬 개발 환경에서는 H2 파일 기반 데이터베이스를, 운영 환경에서는 MySQL을 사용할 수 있도록 설정을 외부화했습니다. API Gateway는 Spring Cloud Gateway WebMVC를 사용합니다.

## 앞으로의 방향

현재까지 구현된 것은 결제 승인의 기본 흐름입니다. 실무 PG 시스템이 되기 위해서는 추가로 다음과 같은 영역이 필요합니다.

승인된 거래를 카드사별로 집계하여 정산하는 배치 처리, 가맹점 인증을 위한 API Key 기반 보안 체계, 거래 통계와 카드사별 승인율을 모니터링하는 관리 기능, 그리고 승인 완료 후 가맹점에 결과를 비동기로 알리는 웹훅 시스템 등이 있습니다. 거래 원장에 이미 정산 상태 필드가 준비되어 있고, 라우팅 서비스가 카드사별로 분리되어 있으므로, 이러한 확장을 자연스럽게 수용할 수 있는 구조적 기반은 갖추어져 있습니다.
